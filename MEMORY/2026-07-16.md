## Robot loop (PD-342 C1) bring-up — container prep + uid-split ownership fixes

Continuing the Robot-loop bring-up (D-055). Merged earlier: **#233** (container prep — `robot` uid 1500 + `gh` + HOME env for the dropped uid) and **#234** (install `gh` from the github.com release tarball, not `cli.github.com`, which the NAS build couldn't resolve). Image verified: `id robot` + `gh 2.63.2` present.

**DB ownership discovery:** the live `dashboard.db` is owned by **`Steve:users` mode 666** (NOT root — the Dockerfile comment was wrong). So lockdown is **`chmod 660` and NO `chown`** (chowning away from the web app's uid breaks the board). Robot uid/gid 1500 is neither owner nor in `users`, so 660 locks it out while web (owner) + root loop keep access.

**Two fixes this exposed → PR #238 (merged?):**
1. **Workspace: shared-checkout worktree → per-ticket clone.** A worktree shares the grounding checkout's `.git`; the root loop creates it, the dropped robot can't write it, and chowning the shared `.git` breaks the root loop's git. Fix: loop clones fresh per ticket + `chown`s the whole clone to the robot uid → robot solely owns the repo it commits in. Fresh clone ⇒ PD-340 pristine automatic. (Matches how Sortie actually worked.)
2. **`checkDbLockedFromCoder` guard was wrong** — it required the *loop* to own the DB (loop=root ≠ Steve), so it'd fail-closed even when perms were right. Reworked to a pure robot-exclusion check (no world access, coding uid≠owner, coding gid≠owning-group-with-access) across the DB + `-wal`/`-shm` sidecars; does NOT require loop ownership (root reaches it via DAC override).
- Also: prompt runs `gh auth setup-git` before push (clone remote is tokenless); loop `mkdir`s `/data/robot-worktrees` itself.

**Key constraint learned (why the loop is root):** dropping the coding session to a lower uid needs `CAP_SETUID` (root-only) — a non-root spawner can't launch a child as a different uid, and chown-to-another-uid is also root-only. So "loop non-root, robot lower" isn't possible with in-process spawn. To truly de-root the loop needs a **two-container split** (loop w/ DB + robot-executor w/o DB, IPC between) → filed **PD-378** (P3, under epic PD-341). For C1, root loop is fine (matches web+worker today; boundary is the egress-hardened network + the robot-below-root drop).

## C2 / PD-343 — fault-aware retry guardrail — built, PR #242

- **New pure engine `faults.ts`**: `classifyFault` / `decideFault` / `preflight` / `backoffMs`. Tiers: **transient** (no-verify/max-turns/flake → retry w/ exp backoff, cap `retryCap=3`), **deterministic** (path-guard/permission patterns OR same-signature repeated `promoteAfter=2`× → park `stuck`, 0 retries), **system-wide** (auth/credit 401/403 → pause loop, excluded from cap = **zero burn**). Signatures normalize paths/numbers/hashes so "same failure" matches across runs.

Loop breaks mid-cycle on pause so no other ticket burns.
- `agent_runs` +`fault_tier`/`fault_signature`/`fault_reason` via **guarded ALTER** (migrates the existing NAS table; pre-C2 rows read back `transient`, signature←status).
- **ask_human** wired end-to-end (it's in the C2 task list): `.robot/ask-human` marker + one prompt para + `readHandoff` → parks `awaiting-human`, run status `ask-human`, not a failure.
- Config knobs (defaulted): `ROBOT_RETRY_CAP`/`PROMOTE_AFTER`/`BACKOFF_BASE_MS`/`BACKOFF_MAX_MS`.
- Out of scope (later slices): fault-tier observability surface = C3; pause/resume UI = C4. C4 consumes `resumeDispatch` + `dispatchPauseState`.
