# 2026-07-07

## Recurring tickets now work (PD-159 merged, #194)

Landed the Task Monitor recurrence logic. **Completing a ticket with `recur_interval` set spawns the next occurrence** into `backlog` — carrying title/body/priority/projectId/assignee/recur_interval, `source:'recur'`, plus a `recurred` event on the completed ticket linking the spawned id. Implemented in `store.ts` `updateTicket` (transition-guarded: `existing.status !== 'completed' && next.status === 'completed'` → idempotent, no double-spawn) and `createTicket` now accepts + persists `recurInterval` (was the gap — `CreateTicketInput` didn't have it), so occurrences **chain**. Fresh occurrence does NOT carry the GitHub issue or `refined` flag.

- This was **Sortie's own in-review PR #194** (branch `sortie/191`, issue #191) — I reviewed it, verified (57 store tests + CI `verify` green), didn't reimplement. Branch protection blocked normal merge → **admin-override squash-merge** (Steve chose it). Deploys to NAS via CI/CD; PD-159 → completed once github-sync picks up the closed issue.
- **Recurrence trigger is `updateTicket` only** — manual board completion (API PATCH) fires it. If GitHub-sync ever sets status via direct SQL (not updateTicket), robot-completed recurring tickets wouldn't respawn; fine for the maintenance use case (manual). Note for future.
- Gotcha: verifying a PR branch in a worktree with symlinked `node_modules` gives a FALSE tsc error — `node_modules/@dashboard/shared` → main's `packages/shared`, so the branch's shared-type changes aren't seen. Trust the PR's own CI, or npm install in the worktree.
