## Epic card priority + inert-status lock (PR #231)

Steve flagged two Epic-card items on the board.
1. **Epic cards never showed priority** — confirmed a *display gap*, not a bug: epics are `agent_tickets` with a real `priority` column, `EpicCard.svelte` just didn't render it. Added an editable priority pill top-right (mirrors `TicketCard`; `.priority` styles copied into `EpicCard.scss` since scoped styles don't cross components), `{done}/{total}` counter moved just to its left.
2. **"Change epic status via edit menu but card stays in lane"** — *working as designed*, not a bug. `deriveEpicLane` (store.ts) computes a non-empty epic's lane **purely from member statuses**, ignoring the epic's own `status` field. Ladder: any member in a queue → In Progress; all done → Completed/Closed; **any member in `backlog` → Backlog**; else → Prioritized. So an epic reaches **Prioritized only when *every* not-yet-done member is Prioritized** (a single backlog member drags the whole epic back). Empty epic → lane follows its own status. Fix (this PR): disable the Status `<select>` in the edit modal for a non-empty epic (`editingEpicWithMembers` derived) with note "Derived from members — prioritize a member to move the Epic."

Web-only (no sensitive paths). `npm run verify` green. Built in worktree `.claude/worktrees/epic-priority` off origin/main — **remove after #231 merges**. No headless-screenshot pass done (Steve didn't ask).
