# 2026-06-29

## SCSS moved into its own files + convention documented

Steve's rule: **SCSS always lives in its own files, never inline in a component.** Implemented + documented.

- Extracted both styled components' `<style>` blocks into sibling SCSS files: `apps/web/src/routes/+page.scss` and `+layout.scss`. Components now reference them via `<style lang="scss" src="./X.scss"></style>`.
- **Gotcha (verified empirically):** `vitePreprocess` **silently ignores** the `src` attribute — the first attempt built green but *dropped* the page's styles entirely (only the layout CSS shipped). Fix: added standalone **`svelte-preprocess`** alongside vitePreprocess in `apps/web/svelte.config.js` (`preprocess: [vitePreprocess(), sveltePreprocess()]`), and installed **`sass`** (which wasn't installed at all — PROJECT.md claimed SCSS but the toolchain lacked it; components were plain CSS).
- Scoping preserved byte-for-byte: layout's scoped hash is identical to the pre-extraction build (`.svelte-12qhfyh`); `:global(...)` tokens/resets still emit unscoped. SCSS nesting (`&:hover`, nested `h2`/`p`) compiles correctly.
- Convention added to **PROJECT.md §5** (Conventions) — chose PROJECT.md over DECISIONS.md since it's a standing style rule, not a one-off architectural decision with weighed alternatives.
- Cold `verify` green (~13.2s, +~2s for the extra preprocessor); `format:check` clean (Prettier handles `.scss`).
