## Filed PD-323 — Sortie auth-fault detector (P2, backlog)

Follow-up to the 2026-07-07 PD-320/#202 incident (see [2026-07-07](2026-07-07.md)): when the bot PAT loses auth, Sortie silently re-dispatches into the 403 wall and burns `max_sessions` across the whole board with no dashboard signal. Filed **PD-323** (prod, P2, backlog) to detect it and pause + surface.

Design context baked into the ticket body (the three constraints that shape it): (1) an auth fault is **system-wide, not per-ticket** — the token 403s for every issue; (2) the error lives **only in the Sortie process log**, not `run_history` (which holds only agent-level failures — `turn_failed`/`context canceled`/`exit 128`); (3) **GitHub can't be the signal path or the stop-lever** — relabeling needs the same broken token, so the report must go via the local board API and "stop retrying" must mean **pause the loop**, not relabel out of the candidate set. Proposed: NAS-side log-watcher → pause dispatch + POST an `authFault` alert to `SystemStatus` (`/api/widgets/task-monitor/system-status`) → red banner in `SystemStatus.svelte`. ## Refine propose_commit — two fixes (decompose "no children" bug + priority gap)

Steve hit **"invalid proposal: no children"** approving a decompose on **PD-273** (agent-worker refine job). Traced via prod events (id 360): the agent's first `propose_commit` (event 824) had `mode:"decompose"` but **no `children` key** — a model slip. The Zod shape in `apps/agent-worker/src/jobs/refine/propose-tool.ts` made `children`/`body` *unconditionally optional regardless of mode*, so the bad call was accepted, persisted, and only caught **much later** at approval time (`store.ts` `approveRefine` → `children.length===0`) as a confusing error to Steve. Re-asking made the agent re-call with children (event 828) → committed (835); Steve self-recovered before I finished.

- **Fix:** enforce the mode↔fields invariant at the **tool boundary**. Added exported, unit-tested `validateProposalShape(proposal)` (`propose-tool.spec.ts`) — decompose needs non-empty `children`, refine_in_place needs `body`. Handler returns `{ isError:true }` with an instructive message so the agent self-corrects **in-turn** instead of deferring a broken proposal to Steve. Zod can't express this (SDK wants a raw shape, not a discriminated union) → guard in the handler.
- **Second gap (Steve asked):** the Refine agent had **no channel to set priority** — `RefineChildProposal`/`RefineProposal` lacked it, so every child landed at unset priority (even PD-273's intended "P3 follow-ons"). Wired **priority end-to-end**: shared types (`priority?: TicketPriority|null`, optional for back-compat w/ already-stored proposals), `propose-tool` `PRIORITY` enum + tool DESCRIPTION, `approveRefine` passes `c.priority ?? null` to `createTicket` and `p.priority` to `updateTicket` (omit = unchanged), `TicketThread.svelte` renders a priority pill in the approval panel. Tests added; `npm run verify` green (357 tests).
- **Lesson (banked):** optional-everything tool schemas defer contradiction detection to approval time — validate mode↔field invariants at the tool call so the *agent* sees the error, not the human.
- **Shipped:** edits were made as uncommitted work on the concurrent session's `feat/site-status-sortie-worker` checkout; isolated them via a patch of only my 7 code files applied into a fresh worktree off `origin/main` (`.claude/worktrees/refine-propose-fix`, real `npm install`), leaving the concurrent session's dirty state untouched. ## Auto-merge bridge was silently dead on the re-fire path — fixed (PR #221)

Steve reported approved+green PRs not auto-merging (PR #216 / D-052 shipped the `sortie-auto-merge.yml` bridge). **Not a container issue** — it's pure GitHub Actions, NAS Sortie containers play no part. Root cause: the bridge declared `check_suite`+`status` triggers to catch "approved before CI went green," but **GitHub suppresses `check_suite`/`status` events that originate from `GITHUB_TOKEN`** (which runs the `ci` workflow) to prevent recursion → those triggers **never fire**. Proof: workflow's entire run history is only `pull_request_review` + `pull_request` events, zero `check_suite`/`status`. So: approve while CI still running → review event sees `mergeStateStatus=UNSTABLE` → skips → the promised re-fire never comes → PR sits approved+CLEAN+unmerged forever. Hit #214 and #217 (both manually squash-merged to clear).

- **Fix (PR #221, branch `fix/auto-merge-workflow-run`, worktree `.claude/worktrees/auto-merge-fix`):** replace dead `check_suite`+`status` with one **`workflow_run`** trigger on `ci` completing (`workflow_run` DOES fire for GITHUB_TOKEN runs — it's the sanctioned chaining exception), gated on `conclusion==success`; resolve `workflow_run.head_sha`→open PR#. Also fixed a **latent bug** in the never-executed SHA→PR resolution: `gh`'s `--jq` flag does NOT accept `--arg` — piped to the runner's real `jq` instead. Amended **D-052** (the `mergeStateStatus==CLEAN` decision is unchanged; only trigger wiring was wrong).
- **`mergeStateStatus` values banked:** `CLEAN` = mergeable+all checks green; `UNSTABLE` = mergeable but a check is pending/failing (non-required); `BLOCKED` = branch-protection unsatisfied (missing required review/check); `DIRTY` = conflict; `BEHIND` = head behind base.
- **Chicken-and-egg to land #221:** the workflow governing #221's own merge is the *buggy* version still on main, so #221 must be **approved AFTER its CI is green** (review event sees CLEAN → merges) or merged by hand. `workflow_run` uses the default-branch version, so the fix only governs PRs opened after #221 lands.
- **Added `pd-runs` / `pd-run-log` to `scripts/pd-aliases.sh`** (the GHA-log commands used to debug this): `pd-runs [workflow] [n]` → `gh run list`; `pd-run-log <id> [regex]` → `gh run view <id> --log | grep -iE`.
