## 17:55 — Task Monitor "hard refresh doesn't reflect closed tickets" — fixed (PD-252, D-043)

**Bug report:** close a ticket, refresh the board → change doesn't show; ~1 min later it appears; hard refresh does nothing.

**Root cause (NOT a fetch/cache bug — the first theory was wrong).** Issue status & labels are GitHub-owned and only land in the SQLite `agent_tickets` table via the server cron [`runGithubSync`](../apps/server/src/widgets/agent-dashboard/github-sync.ts), scheduled `* * * * *` (once a minute). The frontend only ever reads that DB. So a hard refresh re-reads the *not-yet-synced* DB → "does nothing"; the ~1 min is the **cron cadence**, not the 30s client poll. **Key tell:** a hard refresh bypasses the browser HTTP cache, so if it still shows stale data the staleness must be server-side. (An Explore sub-agent's "add `Cache-Control: no-store`" conclusion was a red herring and internally inconsistent — same fetch path for poll & load can't explain "poll updates, refresh doesn't".)

**Fix — on-demand sync trigger (D-043):**
- Server: `requestGithubSync()` in `github-sync.ts` wraps `runGithubSync` with a guard — coalesces concurrent callers onto one in-flight pass, throttles calls within `MIN_ON_DEMAND_INTERVAL_MS = 10_000` (protects GitHub rate limit). Exposed `__resetOnDemandSyncGuard()` for tests.
- Server: `POST /api/widgets/agent-dashboard/sync` in `routes.ts` (503 `NO_READ_TOKEN` when no `GITHUB_READ_TOKEN`; else runs a pass, returns `{outcome: 'ran'|'coalesced'|'throttled'}`). Added `githubReadToken` to `AgentDashboardRouteDeps`.
- Web: `api.syncNow()` + `syncThenLoad()`; board loads DB fast on mount, then fires sync and reloads in place → refresh reflects a just-closed issue within seconds. Added a "Sync now" button (`.sync-btn`).
- Tests: +3 in `routes.spec.ts` (503 / ran / throttled). Full suite green (1070 pass; the 10 "failed files" are pre-existing junk — stale `apps/server/dist/*.spec.js` + the `.claude/worktrees/pd-256-review-bridge` leftover — not my code). tsc + svelte-check + eslint clean.

## PD-256 — Sortie PR-feedback re-work loop was silently dropping feedback (diagnosed + fixed)

**Symptom:** "Request changes" / comments / reviews on in-review `sortie/*` PRs weren't
triggering a re-work — Steve's feedback went unaddressed.

**Root cause (confirmed live):** Sortie's native `reactions.review_comments` only arms its
watch-set at the process-startup **"pending reaction recovery"** pass — one per container
**restart**. An issue that hands off to `sortie:in-review` *between* restarts is never watched,
so a review/comment on its PR is silently dropped until the next restart. Second defect: it only
ever fired on a `CHANGES_REQUESTED` review, never a plain comment or "Comment" review.
- Proof: issue #132 handed off at 03:22, 12m after the 03:10 restart; its 20:44 CHANGES_REQUESTED
  review 17h later → no `reaction_fingerprint` row, no continuation. A manual `docker restart`
  re-armed it (recovery `candidates` 25→29) and it immediately re-worked.
- `reaction_fingerprints` (in `.sortie.db`) is the reaction dedup log; only startup recovery writes it.

**Fix (PR #143, branch `fix/pd-256-review-rework-bridge`, D-042):** in-repo Actions bridge
`.github/workflows/sortie-review-rework.yml` (same pattern as `sortie-conflict-rework.yml`) —
on a trusted human review/comment on a `sortie/*` PR it flips the issue `in-review → queued`;
Sortie re-dispatches, `before_run` reuses the branch, prompt Step 2B reads the feedback.
Triggers broadened to Request-changes/Comment reviews, inline comments, top-level PR comments
(pure Approve excluded). Disabled native `reactions.review_comments`; bumped `max_sessions` 3→5.
**Deploy:** live only once merged to `main`; disabling native reaction + max_sessions needs a
container **recreate** (not restart). Then live-verify per README checklist.
(Numbering reconciled: origin/main has D-041 = Cmd+K/PD-126 and D-042 = PD-256; the PD-252 sync
work — which sat on the older `feat/pd-250-notification-center` branch and had grabbed "D-041" —
