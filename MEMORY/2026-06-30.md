- **CD cloud half:** `.github/workflows/deploy.yml` — on push to `main` (post-merge only, never PR head code), builds `docker/Dockerfile`, pushes `ghcr.io/scolacur/personal-dashboard:latest` + `:<sha>` via GITHUB_TOKEN (packages:write, no PAT). GHA layer cache.

**Key decision — pull-based deploy, NOT self-hosted runner.** NAS is behind home LAN/NAT (no public ingress) AND CPU-weak. Rejected: poll+build on NAS (rebuilds on weak CPU — the TODO RAM/CPU risk), self-hosted Actions runner (untrusted fork-PR code on a *public* repo = hazard). Chose **GHCR + Watchtower pull**: build in free cloud runners, NAS only pulls a finished image. No inbound networking, no untrusted code on NAS.

**Cost = $0 as designed.** Public repo → GHCR public package (unlimited free storage/bandwidth, exempt from the 500MB/1GB free-tier limits that only bill private packages) + Actions minutes unlimited free for public repos. Watchtower is free OSS, self-hosted. Private-package path documented in runbook as fallback.

**Prerequisite flagged:** the app has NEVER been deployed to the NAS (only Sortie has). Watchtower auto-*updates* a running container — runbook Steps 1-4 ARE the first manual app deploy (~1-2 hr, 🧑 Steve-only, NAS/DSM access); every merge after that is hands-off. **The deploy automation immediately earned its keep** — building the image for the first time surfaced a chain of latent bugs that `npm run verify` (which never builds the image) and "the app was never deployed" had hidden:
1. **#10** `apps/web` `prepare: svelte-kit sync` broke prod `--omit=dev` installs (svelte-kit is a devDep) → guarded with `|| true`.
2. **#11** Dockerfile build stage was `node:20-alpine` (musl) but better-sqlite3 ships no musl prebuild + runtime `--ignore-scripts` skipped its binary install.
3. **#11** even on glibc, no prebuilt better-sqlite3 matched the ABI → final fix: **compile in build stage** (add python3/make/g++ toolchain), `npm prune --omit=dev`, **copy the pruned node_modules into runtime** (runtime carries no toolchain, runs no install).
4. **#12** nas compose used `./data`/`./.env` (untracked) → match dev compose's gitignored `../data`/`../.env`.
5. **#13** host **8080 is owned by the gluetun VPN container** → remap app to **`8088:8080`** (LAN URL `http://<nas>:8088`).
6. **#14** DJ library mount path was wrong.

**Durable NAS environment facts (non-obvious; verify before assuming):**
- NAS uses **`docker-compose` v1 (hyphenated)**, NOT the `docker compose` v2 plugin. Run from repo root: `sudo docker-compose -f docker/docker-compose.nas.yml up -d`.
- App published on **host port 8088** (8080 = gluetun). Internal port still 8080.
- DJ library rsync lands at **`/volume1/music/dj-library/tracks`** (mounted `:ro` as `/library`), NOT `dj-library-mirror`. Fixed in both compose files + PROJECT.md §4.
- **GHCR package is public** (repo is public) → NAS pulls with no auth.
- Repo clone on NAS: `/volume1/docker/personal-dashboard/personal-dashboard/`; gitignored `data/` + `.env` at its root.
- Compose changes (port, paths, env) need a **manual `git pull` + `up -d`** on the NAS — Watchtower only auto-updates the *image*, never the compose file.

**Version stamp (#15):** deploy.yml passes git short SHA as `APP_VERSION` build-arg → Dockerfile bakes it into a final `ENV` layer (placed last so only that tiny layer changes per commit, which also moves the digest = Watchtower trigger) → `/api/health` returns `{ok, version}` (`dev` locally).
